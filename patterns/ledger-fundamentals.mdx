These patterns cover the foundations every Canton application depends on: allocating parties correctly and picking the right API surface for your client type.

## Allocate a party safely

<Note>**Problem.** On a fresh Canton install the JSON Ledger API rejects submissions because no party has been allocated.</Note>

**Solution.** Call `/v1/parties/allocate` with a `partyIdHint` you control. Re-using the hint is idempotent; the participant returns the existing identifier.

```bash
curl -X POST $LEDGER/v1/parties/allocate \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"identifierHint": "alice", "displayName": "Alice"}'
```

## Pick between JSON Ledger API and gRPC Ledger API

<Note>**Problem.** Developers default to gRPC out of habit even when a browser-facing dApp would prefer JSON.</Note>

**Solution.** For browser/Node clients use JSON Ledger API v2 (simpler auth, streaming via SSE). For backend services with tight latency requirements use the gRPC API. Both expose the same domain model; the choice is purely about client ergonomics and transport.
