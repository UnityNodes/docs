These patterns help you move from a cryptic Canton error message to a resolution quickly, using community knowledge surfaced by CCPEDIA's search tools.

## Resolve a gRPC error code

<Note>**Problem.** You hit a Canton gRPC error like `UNAVAILABLE` or `TOPOLOGY_TOO_MANY_PENDING_TOPOLOGY_TRANSACTIONS` with no obvious cause.</Note>

**Solution.** Paste the full status into `diagnose_error`. The tool extracts distinctive tokens, including underscore-split parts, and searches across forum posts and GitHub issues for prior resolutions and workarounds.

```bash
// MCP call
diagnose_error({ error_text: "TOPOLOGY_TOO_MANY_PENDING_TOPOLOGY_TRANSACTIONS at submitPartyOnboardingTransaction" })
```

## Diagnose a Canton bug right after a Splice cut

<Note>**Problem.** New Canton errors right after a Splice tag often turn out to be Splice-side, not Canton-side.</Note>

**Solution.** Pull `get_recent_deployments()` to see the freshest releases, then call `get_breaking_changes(from, to)` against the relevant version pair. Forum activity in the 14-day post-release window is usually the most reliable signal for newly surfaced regressions.
