This pattern shows how to implement token transfers using the Canton Token Standard (CIP-0056) instead of writing custom transfer logic from scratch.

## Implement a CIP-0056 token transfer

<Note>**Problem.** Custom transfer choices duplicate logic that the standard already covers, leading to incompatible token implementations.</Note>

**Solution.** Use the Canton Token Standard (CIP-0056) reference flows: `Transfer`, `TransferIntent`, and `TransferReceipt`. Build new tokens by extending the standard, not replacing it. The three-step pattern keeps sender intent, in-flight state, and settlement confirmation cleanly separated.

```daml
-- 1. Sender creates a TransferIntent
template TransferIntent
  with
    sender   : Party
    receiver : Party
    amount   : Decimal
    tokenId  : ContractId Token
  where
    signatory sender

    -- 2. Receiver accepts, triggering the Transfer
    choice Accept : ContractId TransferReceipt
      controller receiver
      do
        archive self
        -- move token custody to receiver ...
        create TransferReceipt with sender; receiver; amount

-- 3. TransferReceipt provides an immutable settlement record
template TransferReceipt
  with
    sender   : Party
    receiver : Party
    amount   : Decimal
  where
    signatory sender, receiver
```

Key points:
- **TransferIntent** is created by the sender and signals intent without moving custody.
- **Transfer** is the in-flight operation that the receiver triggers.
- **TransferReceipt** is the immutable on-ledger settlement record signed by both parties.
- Extend these templates with your token-specific fields rather than creating parallel choice hierarchies.
