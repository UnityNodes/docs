title: "Canton Token Standard: Implementing CIP-0056 Token Transfers"

The Canton Token Standard (CIP-0056) defines the canonical shapes for fungible and non-fungible token operations on the Canton Network. Building tokens that follow the standard means your asset is immediately compatible with CIP-0103 wallets, DEX protocols, and any other Canton application that understands the standard interfaces. Building outside the standard means you duplicate logic that CIP-0056 already solves and create an asset that no standard wallet can move without custom integration.

<Tip>
  Use the CCPEDIA `get_cip_implementation_status` MCP tool to check the current implementation status of CIP-0056 and any CIPs that build on it. The standard is actively developed — confirm you are targeting the correct revision before writing contract code.
</Tip>

<AccordionGroup>
  <Accordion title="Implement a CIP-0056 token transfer">
    **Problem**

    Custom token implementations often duplicate transfer logic that the Canton Token Standard already covers. Each custom transfer choice adds code you must audit, maintain, and explain to integrators. More importantly, non-standard tokens cannot be moved by standard wallets or traded on standard DEX protocols without bespoke bridging code.

    **Solution**

    Use the Canton Token Standard (CIP-0056) reference flows and extend them rather than replacing them. The three core constructs are:

    | Construct | Purpose |
    |---|---|
    | `Transfer` | The completed movement of token ownership from sender to receiver |
    | `TransferIntent` | A declared intention to transfer, enabling multi-step atomic workflows |
    | `TransferReceipt` | The acknowledgement record produced when a transfer completes |

    Build new token types by extending the standard interfaces, not by writing custom transfer choices from scratch:

    ```daml
    -- Extend the standard, don't replace it
    template MyToken
      with
        issuer : Party
        owner  : Party
        amount : Decimal
        -- Standard CIP-0056 fields go here
      where
        signatory issuer, owner

        -- Implement the standard Transfer interface
        interface instance CIP0056.ITransferable MyToken where
          view = CIP0056.TransferableView with ..
          transfer = do
            -- Your custom pre/post transfer logic here
            -- Then delegate to the standard flow
            ...
    ```

    The OpenZeppelin Canton Token Template provides a complete reference implementation. Start from that template when creating a new token type rather than writing from scratch.

    **What you get from following the standard**

    - **Wallet compatibility**: Any CIP-0103 compliant wallet can display and move your token without custom integration
    - **DEX compatibility**: Standard transfer flows work with Canton DEX protocols out of the box
    - **Audit clarity**: Auditors familiar with CIP-0056 can review your token logic without learning a custom protocol
    - **Upgrade path**: When CIP-0056 is updated, you extend the delta rather than rearchitecting your entire token

    **When to extend vs. when to wrap**

    Extend the standard when your token has additional fields or constraints (e.g., expiry, lock conditions, compliance metadata). Wrap the standard when you need to aggregate multiple standard tokens into a higher-level construct. Never bypass the standard transfer flow entirely — if you feel you need to, open a CIP discussion first.
