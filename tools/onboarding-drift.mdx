Getting started on Canton is faster when you have the right guide for your background. CCPEDIA's 7 onboarding and drift tools generate personalised quickstart paths for developers coming from any ecosystem, build topic-driven learning sequences, answer common developer questions, provide structured Canton API references, and expose a unique transparency layer that shows you exactly where curated knowledge bases have drifted from the current live state of the network.

---

### `detect_drift`

Find contradictions between docs, forum posts, and GitHub on a topic. Returns counts of how each surface discusses it, plus the most recent doc-page statement and forum statement, so you can spot mismatches before they confuse a developer. This is distinct from `get_kb_drift`, which compares the Foundation KB to live releases specifically.

<Expandable title="Arguments">
  <ParamField path="topic" type="string" required>
    Topic to cross-check, e.g. `"Daml triggers"`, `"party allocation"`.
  </ParamField>
</Expandable>

<CodeGroup>
```json MCP call
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "detect_drift",
    "arguments": { "topic": "Daml triggers" }
  }
}
```
```bash curl
curl -X POST https://ccpedia.xyz/mcp \
  -H 'content-type: application/json' \
  -H 'accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"detect_drift","arguments":{"topic":"Daml triggers"}}}'
```
</CodeGroup>

### `get_api_reference`

Get a structured reference for one specific Canton Network API: ports, endpoints, services, and documentation links. Covers the JSON Ledger API, gRPC Ledger API, Scan API, Validator API, Token Standard APIs, Admin API, and Splice HTTP API. Use this when the user names one of these Canton APIs directly. For broader doc lookups use `search` or `semantic_search` instead.

### `get_faq`

Returns the top 3 matches. Use this for a single specific question. For a full onboarding path use `get_started_guide`; for API endpoint details use `get_api_reference`.

<Expandable title="Arguments">

### `get_kb_drift`

Report where the Foundation Build-on-Canton KB snapshot (synced daily from `github.com/canton-network-devs/Build-on-Canton-MCP`) diverges from CCPEDIA's live signals — `github_releases` and `mailing_messages`. Returns each drifted `(category, key)` with the KB value, the live value derived right now, the drift age in days, and a severity level (`info`, `warn`, or `stale`).

Use this when a user or agent quotes a Canton SDK, Splice, DPM, or Daml version from the Foundation KB and you need to verify whether it is still current. This cross-source quality audit is unique to CCPEDIA.

<Expandable title="Arguments">
  <ParamField path="refresh" type="boolean">
    If `true`, re-run the validator before returning the report. Default `false` uses the most recent stored findings.
  </ParamField>
</Expandable>

<CodeGroup>
```json MCP call
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "get_kb_drift",
    "arguments": { "refresh": true }
  }
}
```
```bash curl
curl -X POST https://ccpedia.xyz/mcp \
  -H 'content-type: application/json' \
  -H 'accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"get_kb_drift","arguments":{"refresh":true}}}'
```
</CodeGroup>

### `get_radar`

Get the Canton Network Radar — CCPEDIA's curated feed of time-sensitive ecosystem alerts: breaking changes, upgrade deadlines, protocol migrations, and required validator actions. Each item includes a deadline, affected components, and a source link. Filter by level (`critical`, `important`, `ecosystem`, or `all`). Use this when a user asks what is changing, what is about to break, or what deadlines are approaching on Canton.

### `get_started_guide`

Use this when a developer asks how to start building, how to build a dApp, or how to develop on Canton specifically. For a single specific gotcha use `get_faq`; for API details use `get_api_reference`.
